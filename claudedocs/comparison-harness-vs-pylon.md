# Harness vs Pylon — 프로젝트 비교 분석

> 작성일: 2026-03-29

---

## 1. 프로젝트 개요

| 항목 | **Harness** | **Pylon** |
|------|-------------|-----------|
| **한 줄 정의** | 도메인별 에이전트 팀과 스킬을 설계·생성하는 메타 스킬 | AI 멀티에이전트 개발팀 오케스트레이터 (분석→설계→구현→PR) |
| **목적** | "하네스 구성해줘" 한마디로 특정 도메인에 맞는 에이전트 팀 아키텍처, 에이전트 정의, 스킬 파일을 자동 생성 | 사람은 요구사항만 전달 → AI 에이전트 팀이 분석/설계/구현/PR까지 전체 소프트웨어 개발 파이프라인을 수행 |
| **타겟 유저** | Claude Code 사용자 중 다양한 도메인(리서치, 웹개발, 콘텐츠 등)에 에이전트 팀을 구축하려는 사람 | 소프트웨어 개발팀에서 AI 멀티에이전트로 개발 프로세스를 자동화하려는 개발자/팀 리드 |
| **형태** | Claude Code 플러그인 (Markdown 스킬 + 참조 문서) | Go CLI 바이너리 + Claude Code TUI 세션 |
| **GitHub** | revfactory/harness (276★, Apache-2.0) | kyago/pylon (MIT) |
| **생성일** | 2026-03-26 | 2026년 초 (활발한 개발 중) |
| **언어** | Markdown/HTML (코드 없음, 순수 프롬프트 엔지니어링) | Go (실행 가능한 CLI 바이너리) |

### 핵심 차이: "무엇을 만드는가" vs "무엇을 실행하는가"

- **Harness**는 **설계 도구(Architect)**다. 에이전트 팀의 구조, 역할, 스킬을 **생성**하는 것이 목적. 실제 작업 실행은 생성된 에이전트 팀이 별도로 수행한다.
- **Pylon**은 **실행 도구(Orchestrator)**다. 에이전트 팀이 이미 정의되어 있고, 그 팀을 **운영하여 실제 코드를 생산**하는 것이 목적.

## 2. 아키텍처 및 설계 철학

### Harness: 프롬프트 엔지니어링 중심의 메타 스킬

```
사용자 → "하네스 구성해줘" → Claude Code 스킬 트리거
                                    │
                              SKILL.md (6 Phase 워크플로우)
                                    │
              ┌─────────────────────┼─────────────────────┐
              ↓                     ↓                     ↓
     references/              에이전트 정의 생성      스킬 파일 생성
     (패턴 가이드)            (.claude/agents/)     (.claude/skills/)
```

- **Zero-Code 접근**: 코드가 전혀 없다. SKILL.md와 6개 참조 문서(Markdown)만으로 구성
- **6 Phase 워크플로우**: 도메인 분석 → 팀 아키텍처 설계 → 에이전트 정의 → 스킬 생성 → 통합/오케스트레이션 → 검증
- **6가지 아키텍처 패턴**: 파이프라인, 팬아웃/팬인, 전문가 풀, 생성-검증, 감독자, 계층적 위임
- **Claude Code 네이티브**: Claude Code의 Agent Teams API(TeamCreate, SendMessage, TaskCreate)를 직접 활용

### Pylon: Go 바이너리 + LLM-as-Orchestrator

```
사용자 → pylon CLI → Claude Code TUI (LLM-as-Orchestrator)
                          │
           ┌──────────────┼──────────────┐
           ↓              ↓              ↓
    슬래시 커맨드      셸 스크립트      에이전트 23종
    (.pylon/commands/)  (.pylon/scripts/) (.pylon/agents/)
           │
    /pl:pipeline → init → 분석 → 설계 → 구현 → 검증 → PR
```

- **Go 바이너리 기반**: 실제 프로세스 관리, SQLite 저장소, Git 워크트리 격리 등을 Go 코드로 구현
- **파일 기반 상태 관리**: 산출물 존재 = 단계 완료 (artifact-based progress tracking)
- **2계층 에이전트**: 루트 에이전트(PO, PM, Architect 등)와 프로젝트 에이전트(개발자, QA 등)
- **워크스페이스 개념**: 여러 프로젝트(git submodule)를 하나의 워크스페이스에서 관리

### 설계 철학 비교

| 관점 | **Harness** | **Pylon** |
|------|-------------|-----------|
| **접근** | 선언적 (무엇을 만들지 선언) | 명령적 (어떻게 실행할지 제어) |
| **코드 유무** | Zero-Code (순수 Markdown) | Go 바이너리 (수천 줄의 Go 코드) |
| **상태 관리** | Claude Code 내부 상태에 의존 | SQLite + 파일 기반 명시적 관리 |
| **커스터마이징** | 프롬프트 수정으로 무한 도메인 확장 | config.yml + 에이전트 MD 파일 수정 |
| **의존성** | Claude Code만 필요 | Go, git, gh, Claude Code 모두 필요 |
| **런타임 제어** | Claude Code에 위임 | Go 프로세스가 직접 제어 (syscall.Exec) |

## 3. 기술 스택

| 영역 | **Harness** | **Pylon** |
|------|-------------|-----------|
| **구현 언어** | 없음 (Markdown Only) | Go 1.24+ |
| **CLI 프레임워크** | 없음 | Cobra |
| **TUI** | 없음 | charmbracelet/huh (Bubble Tea 생태계) |
| **저장소** | 없음 (파일 시스템) | SQLite (CGO-free, WAL 모드) |
| **검색** | 없음 | FTS5 BM25 (프로젝트 메모리 풀텍스트 검색) |
| **프로세스 실행** | Claude Code 내장 Agent 도구 | syscall.Exec + exec.Command |
| **AI 백엔드** | Claude Code (플러그인으로 동작) | Claude Code CLI (직접 실행) |
| **배포** | 플러그인 마켓플레이스 or 파일 복사 | go install / goreleaser |
| **설정 형식** | plugin.json (플러그인 매니페스트) | YAML (config.yml) |
| **프로젝트 관리** | 없음 (단일 프로젝트 내 동작) | Git submodule 기반 멀티프로젝트 |

## 4. 에이전트 오케스트레이션 방식

### Harness의 접근: 패턴 기반 설계 가이드

Harness는 에이전트를 **직접 실행하지 않는다**. 대신 6가지 아키텍처 패턴을 기반으로 에이전트 팀의 구조를 **설계하고 파일로 생성**한다.

- **실행 모드 선택**: 에이전트 팀(TeamCreate) vs 서브 에이전트(Agent 도구) — 작업 특성에 따라 자동 판단
- **에이전트 분리 기준**: 전문성, 병렬성, 컨텍스트, 재사용성 4축으로 판단
- **데이터 전달**: 메시지 기반(SendMessage), 태스크 기반(TaskCreate), 파일 기반 — 3가지 전략 조합
- **QA 통합**: 점진적 QA(incremental) — 전체 완성 후가 아닌 각 모듈 완성 직후 검증
- **도메인 무관**: 소프트웨어 개발뿐 아니라 리서치, 콘텐츠 제작, 마케팅 등 모든 도메인에 적용 가능

### Pylon의 접근: 파이프라인 기반 실행 엔진

Pylon은 에이전트를 **직접 생성하고 실행한다**. 고정된 파이프라인 워크플로우를 통해 소프트웨어 개발 작업을 자동화한다.

- **고정 파이프라인**: requirement → analysis → architecture → tasks → execution → verification → PR
- **23종 내장 에이전트**: PO, PM, Architect, backend-dev, frontend-dev, QA 등 소프트웨어 개발에 특화된 역할
- **프로세스 격리**: Git worktree로 에이전트 간 파일시스템 격리
- **메모리 시스템**: SQLite + BM25 기반 프로젝트 지식 저장소로 세션 간 학습 유지
- **검증 자동화**: run-verification.sh로 빌드/테스트/린트 교차 검증

### 오케스트레이션 비교 요약

| 관점 | **Harness** | **Pylon** |
|------|-------------|-----------|
| **역할** | 팀 구조를 **설계** | 팀을 **운영** |
| **에이전트 생성** | 도메인 분석 후 동적으로 역할 결정 | 23종 사전 정의된 역할 |
| **실행 방식** | Claude Code Agent Teams API 위임 | Go 프로세스가 직접 관리 |
| **파이프라인** | 6가지 패턴 중 선택 | 고정된 개발 파이프라인 |
| **도메인** | 범용 (모든 도메인) | 소프트웨어 개발 특화 |
| **팀 크기 가이드** | 소(2-3), 중(3-5), 대(5-7) 가이드라인 | max_concurrent 설정으로 제어 |
| **에러 핸들링** | 1회 재시도 → 누락 명시 후 진행 | 검증 실패 시 max_attempts 재시도 |

## 5. 확장성 및 커스터마이징

### Harness

- **도메인 확장**: "~하네스 구성해줘"만으로 새 도메인에 맞는 팀 자동 생성 (리서치, 웹툰, 유튜브, 마케팅 등)
- **아키텍처 패턴 조합**: 6가지 패턴을 상황에 맞게 선택/조합
- **스킬 커스터마이징**: Progressive Disclosure 패턴으로 스킬의 컨텍스트 부하 최적화
- **플러그인 생태계**: Claude Code 마켓플레이스를 통한 배포/설치
- **참조 문서 확장**: references/ 디렉토리에 새 가이드 추가로 패턴 확장
- **harness-100**: 10개 도메인 × 100개 프로덕션 레디 하네스 패키지 제공 (별도 프로젝트)

### Pylon

- **에이전트 커스터마이징**: `.pylon/agents/`에 Markdown 파일 추가/수정으로 에이전트 역할 변경
- **슬래시 커맨드 확장**: `.pylon/commands/`에 새 커맨드 추가
- **셸 스크립트 확장**: `.pylon/scripts/bash/`에 원자적 작업 스크립트 추가
- **도메인 지식**: `.pylon/domain/`에 팀별 컨벤션, 아키텍처, 용어 사전 저장
- **프로젝트 메모리**: BM25 검색 기반 지식 저장소로 세션 간 학습 축적
- **멀티프로젝트**: Git submodule로 여러 프로젝트를 하나의 워크스페이스에서 관리
- **설정 제어**: config.yml로 동시 실행 수, 타임아웃, PR 설정 등 세밀한 제어

## 6. 장단점 및 적합한 사용 시나리오

### Harness

**장점**
- **진입 장벽 제로**: 코드 설치 없이 플러그인 추가만으로 사용 가능
- **도메인 무관**: 소프트웨어 개발 외에도 모든 도메인에 에이전트 팀 구축 가능
- **체계적 설계 방법론**: 6가지 아키텍처 패턴으로 검증된 팀 구조 제공
- **경량**: 의존성이 Claude Code 하나뿐 — 셋업 복잡도 최소
- **재현 가능한 품질**: A/B 테스트에서 평균 품질 +60%, 100% 승률 입증
- **교육적 가치**: 에이전트 팀 설계 패턴을 체계적으로 학습 가능

**단점**
- **실행 엔진 부재**: 팀을 설계만 하고 실행은 사용자/Claude Code에 위임
- **상태 관리 없음**: 파이프라인 진행 상황, 이력, 메모리 등의 영속적 관리 없음
- **소프트웨어 개발 특화 기능 없음**: Git 워크트리 격리, 자동 PR 생성, 빌드/테스트 검증 등 없음
- **멀티프로젝트 미지원**: 단일 프로젝트 범위 내에서만 동작
- **Claude Code 종속**: Claude Code 플러그인 시스템에 완전 의존

**적합한 시나리오**
- 새로운 도메인에 에이전트 팀을 처음 구축할 때
- 비개발 도메인 (리서치, 콘텐츠, 마케팅 등)에 에이전트 자동화를 적용할 때
- 에이전트 팀 아키텍처를 설계하고 최적의 패턴을 탐색할 때
- 빠르게 프로토타입 수준의 에이전트 팀을 만들어 검증할 때

### Pylon

**장점**
- **End-to-End 자동화**: 요구사항 전달부터 PR 생성까지 전체 개발 파이프라인 자동화
- **견고한 런타임**: Go 바이너리로 프로세스 관리, SQLite로 상태 영속화
- **프로젝트 메모리**: BM25 풀텍스트 검색으로 세션 간 학습 지식 축적
- **Git 통합**: 워크트리 격리, 자동 브랜치 관리, PR 생성까지 Git 워크플로우 완전 자동화
- **멀티프로젝트**: 하나의 워크스페이스에서 여러 프로젝트를 통합 관리
- **23종 전문 에이전트**: 소프트웨어 개발의 모든 역할을 사전 정의
- **검증 자동화**: 빌드/테스트/린트 교차 검증으로 품질 보장

**단점**
- **높은 진입 장벽**: Go 1.24+, git, gh, Claude Code 모두 설치 필요
- **소프트웨어 개발 특화**: 비개발 도메인에는 적용 어려움
- **고정된 파이프라인**: 7단계 고정 워크플로우 — 유연한 패턴 선택 불가
- **복잡한 셋업**: 워크스페이스 초기화, 프로젝트 추가, 설정 등 여러 단계 필요
- **Go 의존성**: 빌드/설치에 Go 툴체인 필요

**적합한 시나리오**
- 소프트웨어 개발팀에서 반복적인 개발 작업을 자동화할 때
- 여러 프로젝트를 하나의 워크스페이스에서 통합 관리할 때
- 세션 간 학습/메모리가 중요한 장기 프로젝트에서
- 빌드/테스트/PR까지 완전 자동화된 개발 파이프라인이 필요할 때

## 7. 상호 보완 가능성

두 프로젝트는 경쟁 관계가 아니라 **상호 보완적**이다.

```
Harness (설계)  →  에이전트 팀/스킬 생성  →  Pylon (실행)
   "누구를 만들 것인가"                      "어떻게 운영할 것인가"
```

| 결합 시나리오 | 설명 |
|--------------|------|
| **Harness → Pylon** | Harness로 도메인 맞춤 에이전트를 설계 → Pylon의 `.pylon/agents/`에 배치하여 운영 |
| **Pylon의 에이전트를 Harness로 리디자인** | 기존 23종 에이전트를 Harness의 6가지 패턴으로 재설계하여 최적화 |
| **도메인 확장** | Pylon이 소프트웨어 개발 외 도메인을 지원할 때, Harness로 새 도메인 에이전트 팀 설계 |

## 8. 종합 비교 매트릭스

| 평가 항목 | **Harness** | **Pylon** |
|----------|:-----------:|:---------:|
| 진입 장벽 | 낮음 (플러그인 설치) | 높음 (Go + 다수 의존성) |
| 도메인 범용성 | 높음 (모든 도메인) | 낮음 (소프트웨어 개발 특화) |
| 실행 자동화 | 없음 (설계만) | 높음 (End-to-End) |
| 상태 관리 | 없음 | SQLite + 파일 기반 |
| 메모리/학습 | 없음 | BM25 풀텍스트 검색 |
| Git 통합 | 없음 | 워크트리 + 자동 PR |
| 에이전트 유연성 | 매우 높음 (동적 설계) | 중간 (사전 정의 + 커스텀) |
| 멀티프로젝트 | 미지원 | 지원 (submodule) |
| 커뮤니티/생태계 | 276★, harness-100 패키지 | 개발 초기 |
| 설계 방법론 | 체계적 (6패턴, 4축 분리기준) | 고정 파이프라인 |
| 유지보수 부담 | 거의 없음 (Markdown) | 있음 (Go 코드 + 의존성) |

## 9. 결론

**Harness**와 **Pylon**은 AI 에이전트 오케스트레이션이라는 같은 영역에서 **서로 다른 문제를 해결하는 도구**다.

- **Harness**는 "에이전트 팀을 어떻게 **설계**할 것인가"에 대한 답이다. Zero-code 메타 스킬로, 어떤 도메인이든 체계적인 아키텍처 패턴에 따라 에이전트 팀을 생성한다. 설계의 깊이와 범용성이 강점이다.

- **Pylon**은 "에이전트 팀을 어떻게 **운영**할 것인가"에 대한 답이다. Go 바이너리로 실제 프로세스를 관리하며, 소프트웨어 개발 파이프라인을 End-to-End로 자동화한다. 실행의 완결성과 견고함이 강점이다.

가장 이상적인 활용은 **Harness로 설계하고 Pylon으로 실행**하는 조합이다. Harness의 체계적 설계 방법론으로 최적의 에이전트 팀을 구성하고, Pylon의 견고한 실행 엔진으로 실제 개발 작업을 수행하는 것이다.
