## 2. 접근 방식 타당성 검토 (§3)

### 2.1 접근 A (PO 중심 라우팅) 채택: 적절

채택 근거 5가지 모두 코드베이스로 뒷받침됨:

1. **"도메인 라우팅은 의미 파악 태스크"** — 동의. "AI 트렌드 분석해서 블로그 써줘" 같은 멀티도메인 요청은 키워드로 처리 불가
2. **"기존 PO 프롬프트 인프라 90% 구축"** — 확인. `launch.go:341-361`에 도메인 테이블+파이프라인 설명 완비
3. **"`SuggestWorkflow()` 프로덕션 데드 코드"** — 확인. 확장 기반으로 부적합
4. **"Pylon 설계 철학 일치"** — PO가 단일 진입점으로 자동 라우팅하는 패턴과 일관
5. **"변경 최소화"** — Markdown 수정 중심으로 Go 코드 변경 최소

### 2.2 우려 사항

| 우려 | 심각도 | 설명 |
|------|--------|------|
| 비결정론 | 중간 | 문서에서 인정했으나 **완화 전략 미제시**. 동일 입력에 다른 라우팅 가능 시 디버깅 곤란 |
| 테스트 전략 부재 | 높음 | LLM 기반 라우팅의 **품질 보증 방법이 전혀 논의되지 않음**. 회귀 테스트 불가 |
| 자기참조 패턴 | 낮음 | PO가 routing-decision.json을 작성하고 자신이 읽는 구조. LLM 컨텍스트에서 작동은 하지만 아키텍처적으로 불투명 |

## 3. 구현 계획 검토 (§4)

### 3.1 P0: `pl-pipeline.md` 도메인별 실행 분기 — 적절하나 선행 조건 미충족

**문제**: P0를 실행하려면 §5.2의 미결 사항("비소프트웨어 파이프라인 Steps 상세 설계")이 **먼저** 해결되어야 함.

현재 상태:
- `pl-pipeline.md` Steps 3-9에 도메인 분기를 추가하려면, research/content/marketing 각각의 **구체적 실행 단계**가 필요
- 워크플로우 YAML에 스테이지 순서는 있으나 "어떤 에이전트를 어떤 프롬프트로" 호출하는지 미정의
- 이 결정 없이 P0를 진행하면 또다시 "테이블은 있지만 실행 경로가 없는" 상태가 반복됨

**권고**: §5.2를 P0의 **선행 조건(prerequisite)**으로 명시해야 함.

### 3.2 P1: PO 스킬 목록 보강 — 적절

`launch.go:380-390`에 도메인 라우팅 안내를 추가하는 것은 단순 문자열 추가. 리스크 낮음.

### 3.3 P2: `SuggestWorkflow()` 처리 — 지금 결정 가능

데드 코드가 프로덕션 영향 없으므로 즉시 결정 불필요. 다만 `explore` 워크플로우의 `"research"` 키워드 오매핑은 데드 코드라 현재 무해하지만, 만약 하이브리드 접근(옵션 C) 채택 시 반드시 수정 필요.

## 4. 미결 사항 완전성 검토 (§5)

### 4.1 기존 6개 미결 사항 — 모두 유효하고 잘 분석됨

각 미결 사항의 선택지와 현재 판단이 논리적으로 타당함. 특히:
- §5.1 (routing-decision.json 소비자): 선택지 A+C 판단 합리적
- §5.6 (기존 라우팅 테이블): 선택지 C가 가장 자연스럽다는 판단 동의

### 4.2 누락된 고려사항 — 3건 발견

#### 누락 1: LLM 라우팅 테스트 전략 [심각도: 중간]

접근 A의 최대 약점인 비결정론에 대한 **품질 보증 방안이 완전히 부재**:
- 라우팅 정확도를 어떻게 측정할 것인가?
- 모델 버전 업데이트 시 라우팅 동작 변경을 어떻게 감지할 것인가?
- 최소한 골든 테스트 셋(예: "AI 트렌드 조사" → research, "로그인 버그 수정" → software)이 필요

#### 누락 2: PR #63 멀티도메인 템플릿과의 관계 [심각도: 낮음]

최근 머지된 PR #63("도메인 지식 템플릿 멀티도메인 대응")이 이 이슈에 어떤 영향을 주는지 미언급. 도메인별 템플릿이 PO 라우팅과 어떻게 연계되는지 명확히 할 필요.

#### 누락 3: PO 프롬프트 길이 문제 [심각도: 낮음]

비소프트웨어 파이프라인 실행 단계를 모두 `pl-pipeline.md`에 추가하면 프롬프트가 상당히 길어질 수 있음. 현재 `pl-pipeline.md`는 168줄이며, 3개 도메인 파이프라인을 추가하면 2-3배 증가 가능. 도메인별 별도 스킬 파일 분리 여부 검토 필요.
